# [Multipart Request & Jwt] 유효한 토큰 요청이 403으로 마스킹 (413->403)

## 1. 개요

* **날짜:** 2026.06.01
* **작성자:** 김석환
* **관련 시스템/기능:** 스프링 시큐리티 및 multipart 요청 처리
* **핵심 요약:** 파일 용량 초과 및 에러 코드 마스킹 현상

## 2. 현상 및 상황

* **문제 발생 환경:** 로컬
* **현상:**
    * 초기, PATCH user/profile 요청시, 403 에러코드 발생
    * 마스킹 현상 해결 후, 413 에러코드 발생

## 3. 시도했던 해결 방안

* **가설1:** jwt 페이로드 속, Role 값 문제(USER) -> **결과:** [실패] GET user/profile 요청은 성공
* **가설2:** PATCH 요청 처리 문제 -> **결과:** [실패] POST로 수정해도 403 에러코드 발생
* **가설3:** 타 에러 403으로 가려짐 -> **결과:** [성공] 413 Entity Too Large

## 4. 원인 분석

* 스프링 서버에서 413 에러 발생 후, 스프링이 에러 처리를 위해 내부적으로 /error 경로로 포워딩할 때, 스프링 시큐리티의 해당 주소 명단이
  authenticated로 되어있어, 403 에러코드를 클라이언트측에 반환

## 5. 해결 방안

* /error 경로 허용 및 파일 용량 제한 코드 작성
    * SecurityConfig: .requestMatchers("/error").permitAll() 추가
    * application.yml: 파일 용량 제한 설정(```max-file-size: 5MB max-request-size: 5MB```)
