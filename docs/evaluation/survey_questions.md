# 역량 정리 및 설문 문항 정리

## 개요

* **날짜:** 2026.05.26
* **작성자:** 김석환

## 1. 구현력

### -1. 정의

요구사항을 실제 동작하는 소프트웨어로 구현하고, 유지보수와 확장을 고려한 구조를 설계하며, 발생한 문제를 스스로 해결할 수 있는 능력이다.
단순히 코드를 작성하는 능력이 아니라 설계/구현/디버깅/개선 전 과정을 수행할 수 있는 역량을 의미한다.

### -2. 역량 필요성

개발자의 가장 기본적인 역할은 문제를 코드로 해결하는 것이다. 실무에서는 기능 구현뿐만 아니라 유지보수성과 확장성까지 고려해야 하므로 구현력은
대부분의 개발 직무에서 핵심 역량으로 작용한다.

### -3. 핵심 측정 포인트

- 설계 능력: 요구사항을 코드 구조로 변환할 수 있는가
- 디버깅 능력: 문제의 근본 원인을 추적할 수 있는가
- 코드 품질: 읽기 쉽고 유지보수 가능한 코드를 작성하는가
- 예외 처리: 비정상 상황까지 고려하는가

### -4. 설문 문항

1. 새로운 기능 개발을 시작할 때 가장 먼저 하는 행동은?   
   ① 요구사항을 보면서 바로 코드를 작성하기 시작한다.  
   ② 구현 순서를 간단히 정리한 뒤 개발한다.  
   ③ 필요한 기능과 데이터 흐름을 먼저 설계한 후 구현한다.  
   ④ 향후 확장과 변경 가능성까지 고려하여 구조를 설계한 뒤 구현한다.


2. 예상치 못한 버그가 발생했을 때 주로 어떻게 해결하는가?  
   ① 검색을 통해 비슷한 사례를 찾아 적용한다.  
   ② 로그를 출력하며 문제가 발생한 위치를 추적한다.  
   ③ 로그와 실행 흐름을 분석하여 근본 원인을 찾는다.  
   ④ 원인뿐 아니라 동일 유형의 문제가 재발하지 않도록 구조를 개선한다.


3. 외부 API를 연동하는 기능을 구현할 때 가장 중요하게 생각하는 것은?  
   ① 정상적으로 데이터를 받아오는 기능 구현  
   ② 비정상 데이터에 대한 기본 예외 처리  
   ③ 네트워크 장애·응답 지연 등 다양한 실패 상황 대응  
   ④ 장애 발생 시에도 서비스 영향이 최소화되도록 복구 전략까지 고려


4. 기존 기능은 동작하지만 코드가 매우 복잡합니다. 어떻게 하시겠습니까?  
   ① 동작하므로 그대로 둔다.  
   ② 필요한 부분만 수정한다.  
   ③ 리팩토링하여 구조를 개선한다.  
   ④ 현재 요구사항뿐 아니라 향후 기능 확장을 고려하여 구조를 개선한다.

## 2. 협업

### -1. 정의

여러 개발자와 함께 프로젝트를 수행하면서 정보를 공유하고, 기술적 의사결정을 조율하며, 팀 전체의 생산성을 높이는 능력이다.

### -2. 역량 필요성

현대 소프트웨어는 개인이 아닌 팀 단위로 개발된다. 협업 역량이 부족하면 기술력이 높아도 프로젝트 생산성이 크게 저하될 수 있다.

### -3. 핵심 측정 포인트

- 갈등 해결: 기술 선택 충돌 시 객관적 근거를 통해 조율하는가
- 문서화: 작업 내용을 문서로 공유하는가
- 코드 리뷰: 피드백을 주고받으며 품질 향상에 기여하는가

### -4. 설문 문항

1. 기술적 의견이 충돌했을 때 어떻게 행동하는가?  
   ① 경험이 많은 사람이나 다수 의견을 따른다.  
   ② 자신의 의견을 근거와 함께 설명한다.  
   ③ 각 방안의 장단점을 비교하여 논의한다.  
   ④ 객관적 자료나 간단한 검증 결과를 기반으로 팀 합의를 이끈다.


2. 팀원의 코드를 리뷰할 때 가장 자주 하는 행동은?  
   ① 오류나 오타를 확인한다.  
   ② 개선 가능한 부분을 제안한다.  
   ③ 왜 개선이 필요한지 기술적 이유를 설명한다.  
   ④ 팀 전체 코드 품질 향상 관점에서 대안과 개선 방향을 제시한다.


3. 프로젝트 진행 내용을 공유해야 한다면?  
   ① 구두로 설명  
   ② 채팅으로 전달  
   ③ 문서 정리  
   ④ 문서화 + 의사결정 근거 기록

## 3. 트렌드

### -1. 정의

새로운 기술과 변화하는 개발 환경을 학습하고, 적절한 상황에서 이를 평가하고 활용할 수 있는 능력이다.

### -2. 역량 필요성

IT 산업은 기술 변화 속도가 빠르다. 새로운 기술을 이해하고 적절하게 활용할 수 있는 개발자가 장기적으로 경쟁력을 가진다.

### -3. 핵심 측정 포인트

- 학습 주도성: 새로운 기술을 스스로 탐색하는가
- 정보 탐색 능력: 공식 문서나 원문 자료를 활용하는가
- 기술 선택 능력: 기술의 장단점을 평가하여 선택하는가

### -4. 설문 문항

1. 새로운 기술을 접했을 때 일반적으로 어떻게 학습하는가?  
   ① 개념만 확인한다.  
   ② 블로그·영상 등을 통해 사용 사례를 찾아본다.  
   ③ 공식 문서를 기반으로 직접 사용해본다.  
   ④ 실제 프로젝트 또는 토이 프로젝트에 적용하여 검증한다.


2. 해결 방법이 잘 나오지 않는 문제를 만났을 때 어떻게 정보를 찾는가?  
   ① AI 또는 검색 결과를 그대로 적용한다.  
   ② 커뮤니티와 Stack Overflow 사례를 참고한다.  
   ③ 공식 문서와 기술 문서를 확인한다.  
   ④ 오픈소스 이슈나 소스코드까지 분석하여 원인을 파악한다.


3. 새로운 프레임워크를 도입하자는 의견이 나왔다. 어떻게 판단하는가?
   ① 최신 기술이므로 도입  
   ② 사용 사례 확인  
   ③ 장단점 비교  
   ④ 프로젝트 상황과 유지보수 비용까지 고려

## 4. CS 지식

### -1. 정의

운영체제, 네트워크, 데이터베이스. 자료구조 등 CS의 기본 원리를 이해하고 이를 실제 문제 해결에 활용할 수 있는 능력이다.

### -2. 역량 필요성

프레임워크와 라이브러리는 문제를 추상화하지만 성능 문제나 장애 상황에서는 결국 기본 원리에 대한 이해가 필요하다.

### -3. 핵심 측정 포인트

- 데이터베이스 이해
- 시스템 설계
- 동시성
- 네트워크

### -4. 설문 문항

1. 데이터 조회 속도가 급격히 느려졌을 때 가장 먼저 확인할 것은?  
   ① 서버 자원을 확인한다.  
   ② 쿼리 자체를 점검한다.  
   ③ 인덱스 및 실행 계획을 확인한다.  
   ④ DB 구조, 락, 커넥션 상태까지 종합적으로 분석한다.

2. 많은 사용자가 동시에 접속하는 서비스 설계 시 가장 중요하게 생각하는 것은?  
   ① 서버 성능 향상  
   ② 서버 분산  
   ③ 캐싱 전략 활용  
   ④ 비동기 처리와 시스템 확장성 고려


3. 여러 사용자가 동시에 동일 데이터를 수정할 수 있는 상황에서 가장 우선적으로 고려할 것은?  
   ① 중복 요청 방지  
   ② 예외 처리  
   ③ 동기화 및 경쟁 상태 방지  
   ④ 분산 환경까지 고려한 데이터 정합성 보장


4. 웹 서비스 응답 속도를 개선하기 위한 방법으로 가장 적절한 접근은?  
   ① 전송 데이터 크기 감소  
   ② API 호출 횟수 감소  
   ③ 캐싱 및 CDN 활용  
   ④ 서비스 구조와 통신 방식까지 최적화

## 5. 알고리즘

### -1. 정의

문제를 논리적으로 분석하고 최적의 해결 방법을 설계하며, 효율적인 자료구조와 알고리즘을 선택할 수 있는 능력이다.

### -2. 역량 필요성

복잡한 시스템일수록 문제를 구조적으로 분석하고 효율적으로 해결하는 능력이 중요해진다.

### -3. 핵심 측정 포인트

- 문제 분해 능력: 복잡한 문제를 작은 문제로 나누는가
- 자료 구조 선택: 상황에 맞는 자료구조를 선택하는가
- 복잡도 및 해결 전략 설계: 복잡도를 고려하여, 최적화된 접근 방법을 설계하는가

### -4. 설문 문항

1. 대량 데이터에서 중복 제거와 빠른 탐색이 필요할 때 어떻게 접근하는가?  
   ① 단순 반복문 활용  
   ② 내장 함수 활용  
   ③ 적절한 자료구조(Hash 등) 선택  
   ④ 시간·공간 복잡도를 모두 고려하여 설계


2. 복잡한 요구사항을 구현해야 할 때 주로 어떤 방식으로 접근하는가?  
   ① 구현하면서 해결한다.  
   ② 유사 사례를 참고한다.  
   ③ 문제를 작은 단위로 분해한다.  
   ④ 전체 구조와 예외 상황을 먼저 설계한다.


3. 정렬된 데이터에서 특정 값을 반복적으로 찾아야 한다면?  
   ① 처음부터 순차 탐색한다.  
   ② 제공되는 검색 함수를 활용한다.  
   ③ 이진 탐색과 같은 효율적인 방법을 고려한다.  
   ④ 데이터 규모와 사용 패턴을 분석해 자료구조 자체를 재설계한다.

## 6. 인프라

### -1. 정의

개발한 서비스를 실제 환경에 배포하고 운영하며 안정적으로 유지할 수 있는 능력이다.
로컬 개발을 넘어 서비스 운영 관점의 역량을 의미한다.

### -2. 역량 필요성

좋은 기능도 실제 환경에서 안정적으로 서비스되지 못하면 의미가 없다.
최근에는 개발자에게도 기본적인 운영 및 배포 역량이 요구되고 있다.

### -3. 핵심 측정 포인트

- 배포/자동화: 단순 배포를 넘어 자동화에 대한 경험이 있는가
- 컨테이너 환경: 컨테이너 기술을 활용할 수 있는가
- 모니터링: 장애 대응 경험이 있는가

### -4. 설문 문항

1. 프로젝트를 실제 서비스 환경에 배포할 때 가장 이상적인 방식은?  
   ① 서버에 직접 업로드 후 실행  
   ② 원격 서버에서 수동 배포  
   ③ 컨테이너 기반으로 배포  
   ④ 빌드·테스트·배포가 자동화된 파이프라인 구축


2. 서비스 운영 중 장애를 가장 효과적으로 감지하는 방법은?  
   ① 사용자 제보 확인  
   ② 서버 로그 수동 확인  
   ③ 모니터링 대시보드 활용  
   ④ 자동 알림 및 관측 시스템 구축


3. 개발 환경과 운영 환경 차이로 문제가 발생했다. 어떻게 해결하는가?  
   ① 환경을 수동으로 맞춘다.  
   ② 설정 문서를 만든다.  
   ③ 컨테이너 기반으로 통일한다.  
   ④ 환경 구성을 코드로 관리한다.
